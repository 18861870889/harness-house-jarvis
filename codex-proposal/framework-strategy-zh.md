# Harness House 开源框架方向方案

## 1. 重新定位

Harness House 更适合定位为：

```text
面向多生态智能家居的 AI 编排框架
```

它不应该只做一个“聊天控制家电”的应用，而应该提供一套可复用框架：

- 接入不同生态和厂商设备。
- 把设备能力抽象成统一模型。
- 用 AI 理解自然语言和家庭上下文。
- 生成可解释、可审计、可回滚的执行计划。
- 通过反馈和观察逐步适应常住用户。

核心目标：

```text
从固定 App、固定场景、固定口令，升级到自然语言驱动的家庭智能编排系统。
```

## 2. 当前真实场景

### 2.1 已有设备

现有设备可以分成几类：

| 类别 | 设备 |
| --- | --- |
| 环境控制 | 空调、风扇、窗帘、晾衣杆 |
| 家电控制 | 电视、洗衣机、烘干机、燃气热水器 |
| 清洁维护 | 扫地机器人 |
| 宠物相关 | 猫粮机 |
| 安防感知 | 监控、门窗传感器、玄关人体传感器 |
| 存在感知 | 书房人在传感器、厨房人在传感器 |
| 基础控制 | 智能开关 |

### 2.2 当前痛点

1. App 过多。
   米家和其他厂商 App 各自为政，控制入口分散。

2. 场景绑定死。
   用户必须提前设计固定场景，后续扩展和修改成本高。

3. 口令绑定死。
   传统语音助手通常依赖固定说法，不能稳定理解通用表达。

4. 跨设备意图难表达。
   比如“我要睡了”“厨房有点闷”“帮我准备出门”，这些不是单设备命令，而是状态目标。

5. 家庭习惯不能自然沉淀。
   系统很难逐渐理解常住用户的偏好、作息和例外情况。

## 3. 框架核心能力

Harness House 应该由五个核心层组成。

```text
Connector Layer
  接入米家、Home Assistant、Matter、MQTT、厂商 API、本地协议

Capability Layer
  把不同设备统一抽象成能力，例如 turn_on、set_temperature、open_cover

Context Layer
  管理房间、人在状态、时间、天气、门窗状态、设备状态、家庭成员偏好

Agent Layer
  理解自然语言，生成计划，处理歧义，执行安全策略

Evolution Layer
  从反馈、历史和用户修正中学习，但学习结果必须可查看、可撤销、可确认
```

这五层是开源框架价值所在。上层 UI 可以换，底层设备生态也可以换，但中间的家庭语义和 AI 编排逻辑可以复用。

## 4. 接入路线

### 4.1 第一阶段：Home Assistant 作为桥接层

建议第一阶段不要直接同时接米家和一堆厂商 App。更务实的做法是：

```text
Harness House -> Home Assistant -> 米家/其他厂商设备
```

这样可以先验证 AI 编排框架，而不是把精力消耗在每个厂商的接入细节上。

这一阶段重点：

- 从 Home Assistant 同步设备、房间、状态。
- 建立 Harness House 自己的统一设备能力模型。
- 支持自然语言控制和计划生成。
- 通过审计日志记录每次 AI 决策。

### 4.2 第二阶段：增加原生连接器

当核心 agent 跑通后，再逐步增加：

- 米家连接器。
- MQTT 连接器。
- Matter 连接器。
- 厂商云 API 连接器。
- 局域网协议连接器。

连接器必须遵守统一接口，不能把厂商逻辑泄漏到 agent 核心里。

## 5. 统一能力模型

设备不要直接按品牌建模，而要按能力建模。

例如：

```text
智能开关:
  - turn_on
  - turn_off
  - get_state

空调:
  - turn_on
  - turn_off
  - set_temperature
  - set_mode
  - set_fan_speed

窗帘:
  - open
  - close
  - set_position

人在传感器:
  - get_presence
  - get_last_seen

猫粮机:
  - dispense_food
  - get_food_level
  - get_last_feed_time

燃气热水器:
  - get_state
  - set_temperature
  - turn_on
  - turn_off
```

这样用户说“厨房有点闷”，agent 可以推理：

```text
厨房有人 -> 当前温湿度/空气状态 -> 可用设备有风扇/窗户/空调/排气设备 -> 生成候选方案
```

而不是只能匹配固定口令。

## 6. 设备风险分级

你现在的设备里，风险差异很大。建议一开始就内置风险分级。

| 风险 | 设备/动作 | 默认策略 |
| --- | --- | --- |
| 低 | 智能开关、风扇、窗帘、电视基础控制、晾衣杆 | 可直接执行，保留日志 |
| 中 | 空调温度、扫地机器人、洗衣机、烘干机、猫粮机 | 视上下文执行，必要时确认 |
| 高 | 燃气热水器、门窗相关动作、监控隐私相关操作 | 默认确认 |
| 敏感 | 摄像头画面、家庭成员习惯、人在状态历史 | 严格权限和审计 |

特别注意：

- 猫粮机不是普通低风险设备，过量投喂会影响宠物健康。
- 燃气热水器要按高风险处理。
- 监控和人在传感器涉及隐私，不应让 AI 无限制总结个人行为。

## 7. 自然语言理解目标

系统需要支持三类表达。

### 7.1 明确命令

```text
把书房风扇打开。
卧室空调调到 25 度。
把窗帘关上。
```

输出应该是确定设备和动作。

### 7.2 模糊目标

```text
厨房有点闷。
我要睡了。
准备看电影。
我快到家了。
```

输出应该是一个计划，而不是单条命令。

### 7.3 纠正和偏好

```text
以后晚上看电视不要把灯关太暗。
书房有人时别让扫地机器人进去。
猫粮机每天最多自动加一次，其他情况先问我。
```

这类输入不只是控制命令，而是规则和偏好更新。

## 8. Agent 自进化设计

所谓“自进化”不能理解为 agent 私自改规则、私自控制家庭。更安全的定义应该是：

```text
系统能从用户反馈和历史行为中形成候选偏好，并在用户确认后固化为规则。
```

### 8.1 自进化闭环

```text
观察 -> 归纳 -> 生成候选偏好 -> 请求确认 -> 生效 -> 持续评估 -> 可撤销
```

### 8.2 可学习内容

适合学习：

- 常用场景偏好。
- 不同房间的舒适温度。
- 不同时间段的灯光亮度。
- 常住用户的起居习惯。
- 设备联动偏好。
- 禁止自动控制的例外。

不适合自动学习：

- 门锁权限。
- 燃气设备自动开启。
- 监控隐私分析。
- 涉及人身安全的规则。
- 宠物投喂的极限规则。

### 8.3 学习结果必须可见

需要有一个“记忆与规则中心”：

- AI 记住了什么。
- 这些记忆来自哪次对话或行为。
- 当前是否启用。
- 最近触发了几次。
- 能否暂停、删除、回滚。

### 8.4 从反馈中学习

典型例子：

```text
用户：我要睡了。
AI：准备关闭客厅电视，关闭客厅灯，卧室空调设为 25 度，关闭窗帘。
用户：窗帘别关，我喜欢留一点缝。
AI：明白。是否以后睡眠场景都把卧室窗帘保留 20%？
用户：可以。
```

这时系统新增一条可审计偏好：

```json
{
  "type": "scene_preference",
  "scene": "sleep",
  "target": "bedroom_curtain",
  "preference": "set_position",
  "value": 20,
  "source": "user_confirmation",
  "status": "active"
}
```

## 9. Planner 设计

Agent 不应该直接执行自然语言。推荐流程：

```text
用户输入
  -> 意图识别
  -> 上下文检索
  -> 设备候选匹配
  -> 计划生成
  -> 风险评估
  -> 用户确认或自动执行
  -> 设备执行
  -> 结果总结
  -> 反馈学习
```

计划对象示例：

```json
{
  "user_goal": "我要睡了",
  "scene": "sleep",
  "steps": [
    {
      "device": "living_room_tv",
      "action": "turn_off",
      "risk": "low",
      "reason": "睡眠场景通常关闭客厅电视"
    },
    {
      "device": "bedroom_ac",
      "action": "set_temperature",
      "value": 25,
      "risk": "medium",
      "reason": "用户偏好睡眠温度为 25 度"
    }
  ],
  "requires_confirmation": true
}
```

## 10. 开源框架的项目边界

为了成为开源框架，项目需要把“产品功能”和“框架能力”分开。

### 10.1 框架核心

- 设备能力模型。
- 连接器接口。
- 上下文模型。
- 意图 schema。
- 计划 schema。
- 策略引擎。
- 执行引擎。
- 记忆和偏好系统。
- 审计日志。

### 10.2 参考应用

- Web 控制台。
- AI 对话面板。
- 设备状态面板。
- 执行计划预览。
- 规则和记忆中心。
- 审计日志查看器。

### 10.3 插件生态

后续可开放：

- 设备连接器插件。
- 场景模板插件。
- 传感器解释插件。
- LLM provider 插件。
- 本地模型插件。
- 风险策略插件。

## 11. MVP 建议

第一版不要追求“全自动家庭大脑”。建议 MVP 是：

```text
多设备接入 + 自然语言计划 + 安全执行 + 反馈学习雏形
```

具体范围：

1. Home Assistant 接入。
2. 同步设备、房间、状态。
3. 内置你现有设备类型的 capability schema。
4. Web 控制台。
5. AI 对话输入。
6. 生成结构化执行计划。
7. 风险分级和确认机制。
8. 执行日志。
9. 简单偏好记忆，例如“以后睡觉时窗帘留 20%”。
10. 规则中心可查看和删除记忆。

## 12. 第一批重点场景

### 12.1 睡眠场景

输入：

```text
我要睡了。
```

可能计划：

- 关闭客厅电视。
- 关闭非必要灯光和开关。
- 卧室空调设为偏好温度。
- 关闭或半关窗帘。
- 确认门窗传感器状态。
- 不主动操作燃气热水器，除非用户明确要求。

### 12.2 回家场景

输入：

```text
我快到家了。
```

可能计划：

- 打开玄关或客厅灯。
- 根据温度开启空调或风扇。
- 打开窗帘或设置到白天状态。
- 不自动解锁门，除非有更严格的本地确认机制。

### 12.3 离家场景

输入：

```text
我要出门了。
```

可能计划：

- 关闭不必要开关。
- 关闭电视。
- 关闭风扇。
- 检查门窗传感器。
- 暂停高风险设备。
- 提醒洗衣机/烘干机/燃气热水器状态。

### 12.4 厨房舒适场景

输入：

```text
厨房有点闷。
```

可能计划：

- 判断厨房是否有人。
- 如果有风扇或排气设备，打开。
- 如果窗户状态可检测，结合门窗传感器建议开窗或关窗。
- 不涉及燃气设备自动操作。

### 12.5 宠物喂食场景

输入：

```text
给猫加点粮。
```

可能计划：

- 查询今日喂食次数。
- 检查最近一次喂食时间。
- 如果超过安全阈值，请求确认。
- 记录投喂事件。

## 13. 技术路线建议

### 13.1 后端

- Python FastAPI。
- Pydantic 定义所有 schema。
- SQLite 作为 MVP 存储。
- 后续可切 PostgreSQL。

### 13.2 Agent

- LLM 只输出结构化 JSON。
- 所有 JSON 必须 schema validation。
- 设备匹配、风险评估、执行都由确定性代码完成。
- 支持多模型 provider，不能绑定单一模型。

### 13.3 前端

- Web-first。
- 首页就是家庭控制台。
- 必须有执行计划预览。
- 必须有规则/记忆中心。
- 必须有审计日志。

### 13.4 测试

必须内置 fake home adapter，用来模拟：

- 智能开关。
- 空调。
- 窗帘。
- 电视。
- 人体/人在传感器。
- 猫粮机。
- 燃气热水器。

开源框架不能要求贡献者真的拥有这些设备才能开发。

## 14. 我建议的下一步

下一步不是马上写 UI，而是先定义核心协议：

1. `Device` schema。
2. `Capability` schema。
3. `Intent` schema。
4. `Plan` schema。
5. `Command` schema。
6. `Policy` schema。
7. `Memory` schema。
8. `Connector` interface。

这些定义清楚后，Harness House 才会像一个框架，而不是一个临时拼出来的智能家居 App。

## 15. 一句话版本

Harness House 应该做成：

```text
一个开源的、跨生态的、以自然语言和家庭上下文为入口的智能家居 AI 编排框架；
它能连接多种设备生态，生成安全可解释的执行计划，并通过用户确认和反馈逐步适应常住用户。
```
