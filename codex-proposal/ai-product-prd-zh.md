# Harness House AI 产品 PRD

> 版本：v0.1  
> 定位：开源智能家居 AI 控制框架  
> 目标：自然语言控制全屋设备，2 秒内返回执行/确认/失败结果，并通过可审计记忆逐步适应常住用户。

## 1. 产品概述

### 1.1 产品名称

Harness House

### 1.2 一句话定义

Harness House 是一个开源智能家居 AI 控制框架，用自然语言理解用户意图，将多生态智能家居设备抽象成统一能力，并在安全策略约束下快速执行。

### 1.3 产品愿景

消灭智能家居的 App 地狱、口令地狱和死板场景配置，让家能够听懂通用语言，并逐步适应常住用户的习惯。

### 1.4 产品形态

第一阶段产品不是一个完整的 autonomous home agent，而是：

```text
Fast AI Home Control Runtime
```

核心能力：

- 常见命令快速执行。
- 模糊指令由 AI 解析成结构化计划。
- 中高风险动作必须确认或阻断。
- 自进化在后台产生候选偏好，用户确认后生效。
- 通过 Home Assistant 优先接入真实设备，同时保留连接器扩展能力。

## 2. 背景与问题

### 2.1 当前用户场景

用户已有设备：

- 智能开关
- 空调
- 风扇
- 窗帘
- 电视
- 燃气热水器
- 玄关人体传感器
- 书房/厨房人在传感器
- 门窗传感器
- 猫粮机
- 晾衣杆
- 扫地机器人
- 洗衣机
- 烘干机
- 监控

当前生态：

- 米家
- 其他厂商 App
- 多个设备入口割裂

### 2.2 核心痛点

| 痛点 | 说明 |
| --- | --- |
| App 过多 | 不同设备分散在不同 App 中 |
| 固定口令 | 传统语音控制要求用户记住固定说法 |
| 场景死板 | 场景绑定固定条件，临时调整成本高 |
| 跨设备意图难表达 | “我要睡了”“厨房有点闷”不是单设备命令 |
| 无持续适应 | 系统不能自然学习常住用户偏好 |
| AI 执行风险 | AI 误解后可能错误控制设备 |

### 2.3 产品机会

智能家居 AI 产品的机会不是“聊天”，而是：

```text
把自然语言转成安全、可解释、可执行、可回滚的家庭控制计划。
```

## 3. 目标与非目标

### 3.1 产品目标

1. 用户可以用通用自然语言控制常见智能家居设备。
2. 明确低风险命令在 2 秒左右返回执行结果。
3. 模糊命令最多一次大模型调用，返回结构化计划或确认请求。
4. 所有设备动作经过确定性安全策略。
5. 用户能看到 AI 做了什么、为什么做、结果如何。
6. 系统能从用户反馈中生成候选偏好，但必须用户确认后才生效。
7. 架构上支持多连接器，P0 以 Home Assistant 为真实设备入口。

### 3.2 非目标

P0 不做：

- 全自动家庭管家。
- AI 直接调用设备 API。
- 高风险设备自动执行。
- 复杂多用户画像。
- 原生接入所有厂商。
- 强化学习实时接管。
- 长链路多轮 agent 推理。

## 4. 目标用户

### 4.1 主要用户

有一定技术能力、拥有多种智能家居设备、愿意自部署或调试开源项目的用户。

### 4.2 次要用户

智能家居爱好者、Home Assistant 用户、开源贡献者。

### 4.3 当前非目标用户

完全不懂部署、不愿配置设备、期待消费级开箱即用体验的普通用户。

## 5. 核心用户故事

### 5.1 明确设备控制

```text
作为用户，
我希望说“打开书房风扇”，
系统能快速识别书房风扇并执行，
让我不需要打开设备 App。
```

验收标准：

- 命中 Fast Path。
- 2 秒内返回执行结果。
- 记录审计日志。

### 5.2 模糊意图控制

```text
作为用户，
我希望说“厨房有点闷”，
系统能结合厨房人在传感器和可用设备，
建议或执行合适动作。
```

验收标准：

- 使用上下文快照。
- 大模型只输出结构化 intent/plan。
- 如果动作风险不高且置信度足够，可执行。
- 如果不确定，返回确认请求。

### 5.3 场景控制

```text
作为用户，
我希望说“我要睡了”，
系统能生成睡眠场景计划，
包括关闭电视、调整空调、处理窗帘等动作。
```

验收标准：

- 生成多步骤计划。
- 中风险步骤可配置是否确认。
- 高风险动作不自动执行。
- 用户可以确认、取消或修改。

### 5.4 偏好学习

```text
作为用户，
我希望系统逐渐记住“睡觉时窗帘留 20%”，
但这个记忆必须先问我是否确认。
```

验收标准：

- 后台从日志发现候选偏好。
- 明确询问用户是否固化。
- 记忆中心可查看、暂停、删除。

## 6. 产品原则

1. 快速响应优先。
   常见指令不能每次都依赖大模型。

2. AI 只理解和规划，不直接执行。
   设备执行必须由确定性代码完成。

3. 安全策略优先于智能程度。
   宁可多问一次，也不能错误执行高风险动作。

4. 记忆必须可见。
   用户要知道系统记住了什么、为什么记住、何时触发。

5. 连接器可替换。
   Home Assistant 是第一接入层，但不是唯一地基。

## 7. 核心体验流程

### 7.1 低风险明确命令

```text
用户：关客厅灯
系统：已关闭客厅灯
```

链路：

```text
User Command
-> Gateway
-> Command Router
-> Fast Path Parser
-> Plan Validator
-> Safety Gate
-> Device Executor
-> Home Assistant
-> Result
```

### 7.2 模糊命令

```text
用户：太亮了
系统：已将当前所在书房灯光调低到 35%
```

链路：

```text
User Command
-> Gateway
-> Command Router
-> LLM Intent Parser
-> Plan Validator
-> Safety Gate
-> Device Executor
-> Result
```

### 7.3 需要确认的命令

```text
用户：打开燃气热水器
系统：这是高风险设备。是否确认打开燃气热水器？
```

链路：

```text
User Command
-> Intent Parser
-> Safety Gate
-> Confirmation Required
```

## 8. 功能需求

### 8.1 P0 功能

| ID | 功能 | 描述 | 验收标准 |
| --- | --- | --- | --- |
| P0-01 | Interaction Gateway | 接收 Web/API/Bot 输入 | 能生成统一 command request |
| P0-02 | Context Snapshot | 后台维护房间、设备、人在状态 | 主链路读取耗时 < 30ms |
| P0-03 | Fast Path Parser | 解析常见明确命令 | 无 LLM，命中后可执行 |
| P0-04 | LLM Intent Parser | 模糊语言转结构化 JSON | 单次 no-think 调用，schema 可校验 |
| P0-05 | Plan Validator | 校验 intent/plan/command | 非法输出不能执行 |
| P0-06 | Safety Gate | 风险分级和确认策略 | 高风险默认确认或阻断 |
| P0-07 | Device Executor | 下发命令和收集结果 | 返回 accepted/executed/failed |
| P0-08 | Home Assistant Connector | 接入真实设备 | 支持状态查询和服务调用 |
| P0-09 | Audit Log | 记录每次动作 | 包含用户输入、计划、结果、风险 |
| P0-10 | Memory Center MVP | 查看已确认偏好 | 支持启用、停用、删除 |

## 9. AI 能力设计

### 9.1 AI 在系统中的职责

AI 负责：

- 理解自然语言。
- 识别模糊意图。
- 将意图转换为结构化 JSON。
- 生成计划说明。
- 根据历史偏好辅助计划。

AI 不负责：

- 直接调用设备 API。
- 绕过安全策略。
- 自行修改规则。
- 自行开启高风险设备。
- 在未授权情况下分析隐私数据。

### 9.2 模型调用策略

```text
Fast Path 命中：不调用模型。
Fast Path 未命中：调用一次 no-think LLM。
LLM 超时：返回保守确认或失败原因。
高风险动作：即使 LLM 高置信，也必须确认。
```

### 9.3 LLM 输入

输入必须短小：

- 用户原文。
- 当前上下文快照。
- 设备 capability 摘要。
- 已确认偏好摘要。
- 输出 schema。

禁止把全量日志、全量设备历史、长文档塞给模型。

### 9.4 LLM 输出

只允许结构化 JSON：

```json
{
  "intent": "control_device",
  "confidence": 0.86,
  "targets": [
    {
      "room": "study",
      "device_type": "fan"
    }
  ],
  "actions": [
    {
      "capability": "turn_on"
    }
  ],
  "needs_confirmation": false,
  "reason": "用户表达了打开书房风扇的明确意图"
}
```

如果模型输出不符合 schema：

- 不执行。
- 返回解析失败或转确认。
- 记录失败样本，用于后续评测。

## 10. 设备风险策略

| 风险级别 | 设备/动作 | 默认策略 |
| --- | --- | --- |
| 低 | 开关、灯、风扇、电视基础控制 | 可自动执行 |
| 中 | 空调、窗帘、晾衣杆、扫地机器人、洗衣机、烘干机、猫粮机 | 根据上下文执行或确认 |
| 高 | 燃气热水器、门窗相关关键动作 | 默认确认 |
| 敏感 | 监控、人在状态历史、家庭成员习惯 | 权限控制和审计 |
| 禁止自动 | 危及人身/财产安全的动作 | 阻断 |

特别规则：

- 猫粮机需要投喂频率限制。
- 燃气热水器默认高风险。
- 监控不能被 AI 默认用于隐私分析。
- 人在传感器只能作为即时上下文，不应无限期生成个人行为画像。

## 11. 性能要求

### 11.1 响应 SLA

| 场景 | 目标 |
| --- | --- |
| Fast Path 明确低风险命令 | 150-800ms |
| LLM Path 模糊低风险命令 | 900-1900ms |
| 需要确认的中高风险命令 | 2s 内返回确认请求 |
| 设备物理动作较慢 | 2s 内返回已下发，后续异步更新 |

### 11.2 2 秒定义

2 秒内返回：

- 已执行
- 已下发
- 需确认
- 失败原因

不要求所有物理设备在 2 秒内完成动作。

## 12. 数据与隐私

### 12.1 必须记录

- 用户输入
- 解析结果
- 执行计划
- 风险等级
- 是否确认
- 下发命令
- 执行结果
- 失败原因

### 12.2 谨慎记录

- 人在状态历史
- 监控相关事件
- 家庭成员习惯
- 宠物喂食记录

### 12.3 用户可控

用户必须能：

- 查看偏好记忆。
- 删除偏好记忆。
- 暂停某条规则。
- 查看最近执行记录。
- 禁止某些设备被 AI 自动控制。

## 13. 评测体系

AI 产品必须有评测集，而不是只靠主观体验。

### 13.1 意图解析评测

样本类型：

- 明确命令
- 模糊命令
- 查询命令
- 场景命令
- 高风险命令
- 否定/取消命令
- 歧义命令

指标：

| 指标 | 目标 |
| --- | --- |
| intent 准确率 | P0 >= 90% |
| 设备匹配准确率 | P0 >= 90% |
| 高风险识别召回率 | P0 >= 99% |
| 非法 JSON 率 | < 1% |
| 需确认判断准确率 | P0 >= 95% |

### 13.2 延迟评测

必须记录：

- gateway latency
- context read latency
- router latency
- LLM latency
- validation latency
- safety latency
- executor latency
- connector latency

### 13.3 安全评测

必须覆盖：

- 燃气热水器误操作。
- 监控隐私请求。
- 猫粮机频繁投喂。
- 用户模糊表达导致的错误设备匹配。
- “不要关灯”等否定语义。
- “除了书房都关灯”等范围语义。

## 16. 开放问题

1. P0 主入口优先 Web、微信 Bot 还是 API？
2. 当前设备是否都能先接入 Home Assistant？
3. 是否允许空调类中风险动作默认执行？
4. 猫粮机的投喂上限如何设置？
5. 监控是否只允许状态查询，不允许内容分析？
6. 第一版是否需要中文优先，英文后置？

## 17. 推荐结论

Harness House 的 AI 产品 PRD 应以低延迟和安全执行为中心。

最终原则：

```text
常见命令不用大模型。
模糊命令最多一次大模型。
设备执行必须经过确定性安全策略。
自进化不进入主链路，只做后台候选偏好。
所有 AI 输出必须结构化、可校验、可审计、可回滚。
```
