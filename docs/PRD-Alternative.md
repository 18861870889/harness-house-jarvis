## 1. 产品概述

### 1.1 核心隐喻

不是"AI管家"（中心控制器），而是**"AI管道工"**——家里的事件像水流一样经过管道，AI在管道的阀门处做决策，而不是站在门口指挥一切。

### 1.2 核心洞察

原方案假设"用户说指令 → AI理解 → AI执行"，这是**请求-响应模式**。

但实际智能家居的绝大多数操作是**事件驱动的**：
- 传感器检测到人来了 → 灯开了
- 温度超阈值 → 空调开了
- 时间到了 → 窗帘关了

用户手动说指令的场景只占10-20%。真正的价值在于**让AI接管事件驱动的80%**，同时让手动指令成为"覆盖层"——当AI的自动决策不符合意图时，用户一句话纠正，AI立刻学习。

## 2. 事件驱动架构

### 2.1 核心流水线

```
Event Source ──→ EventBus ──→ Router ──→ Handler Pipeline ──→ Action

  传感器事件        │          │           │                    │
  (人在/温度/门窗)   │          │           │                    │
                    │    ┌─────▼─────┐  ┌──▼───┐          ┌──▼───┐
  定时事件           │    │ LLM Agent │  │Rule  │          │Device│
  (cron/日出日落)    │    │ (消歧+决策)│  │Engine│          │Bus   │
                    │    └─────┬─────┘  └──┬───┘          └──┬───┘
  用户指令           │          │           │                    │
  (微信/语音)        │          │           │                    │
                    │    ┌─────▼───────────▼─────┐             │
  设备状态变更       │    │  Safety Guardian      │             │
  (开/关/异常)       │    │  (分级拦截)            │             │
                    │    └───────────┬───────────┘             │
                    │                │                          │
                    │         ┌──────▼──────┐                   │
                    │         │ RL Learner  │                   │
                    │         │ (奖励/惩罚)  │                   │
                    │         └──────┬──────┘                   │
                    │                │                          │
                    └────────────────┼──────────────────────────┘
                                     │
                              ┌──────▼──────┐
                              │  TimescaleDB │
                              │  (全量记录)  │
                              └─────────────┘
```

**关键区别**：LLM Agent不是中心控制器，而是管道中的一个智能节点。当事件流过时：
- 简单事件（传感器→灯开）直接走Rule Engine，不需要LLM
- 模糊事件（"太亮了"）才路由到LLM Agent做意图理解
- 所有动作都经过Safety Guardian拦截

### 2.2 路由策略

```python
def route_event(event):
    if event.type in KNOWN_RULES:
        # 已有确定性规则 → 直接走Rule Engine，0延迟
        return RuleEngine.handle(event)
    elif event.type == "user_command":
        # 用户指令 → 路由到LLM消歧
        confidence = estimate_confidence(event)
        if confidence > 0.7:
            return LLMAgent.handle(event)  # 高置信 → 直接执行
        else:
            return LLMAgent.handle_with_confirm(event)  # 低置信 → 确认后执行
    elif event.type == "sensor_trigger":
        # 传感器事件 → 检查是否有RL学到的映射
        learned_action = RL.lookup(event)
        if learned_action:
            return learned_action  # RL已学会 → 自动执行
        else:
            return RuleEngine.handle(event)  # RL未学会 → 走默认规则
```

**好处**：
- 90%的日常操作走Rule Engine，**零LLM调用成本，零延迟**
- 只有模糊指令才调LLM，**API成本降到1/10**
- 本地LLM优先处理模糊指令，**隐私不泄露**
- RL Learner逐渐学会映射，**LLM调用越来越少**

## 3. 强化学习进化机制

### 3.1 核心思想

不是"发现模式→建议场景→等用户确认"（被动模式），而是：
- AI主动尝试决策 → 用户反馈=奖励信号 → RL模型更新 → 下次更准

**类比训练狗**：不是先分析狗的行为模式再建议规则，而是狗做了动作，你给反馈（奖励/惩罚），狗逐渐学会正确行为。

### 3.2 奖励信号设计

| 用户行为 | RL奖励值 | 含义 |
|---|---|---|
| 用户说"嗯"/"好"/"开" | +1.0 | AI决策正确，强化此映射 |
| 用户不干预（沉默接受） | +0.5 | 轻度正反馈，AI决策可能正确 |
| 用户纠正"不是书房，是客厅" | -0.5 + 新映射 | AI消歧有偏差，更新+惩罚 |
| 用户说"不对"/"停"/"取消" | -1.0 | AI决策错误，大幅惩罚 |
| 用户主动说指令（而非AI自动） | 0 | 中性信号，记录但不惩罚 |

### 3.3 RL状态空间

```python
state = {
    "location": 当前人在传感器推断的位置,  # 书房/客厅/厨房/玄关/未知
    "time_slot": 当前时间段,                # morning/afternoon/evening/night
    "temperature_indoor": 室内温度,
    "temperature_outdoor": 室外温度(天气API),
    "light_level": 当前区域光照强度,
    "day_of_week": 星期几,
    "season": 季节(春夏秋冬),
    "recent_actions": 最近5分钟的操作序列,
}
```

### 3.4 RL动作空间

```python
action = {
    "device": 目标设备,
    "operation": 操作类型(on/off/set_value),
    "value": 操作值,
    "confidence": RL模型输出的置信度,
}
```

**置信度 < 0.6 → 自动转为确认模式**（不对L2/L3设备低置信自动执行）

### 3.5 进化路径

```
Week 1-2 (冷启动):
  · 所有决策走Rule Engine + LLM消歧
  · RL Learner只做记录，不做决策
  · 用户反馈积累初始训练数据

Week 3-4 (学习期):
  · RL Learner开始在L1设备上做决策尝试
  · 低置信(<0.6)仍走LLM确认
  · 高置信(>0.6)的L1设备直接执行

Month 2+ (成熟期):
  · RL在L1设备上达到>80%准确率 → 自动执行
  · RL在L2设备上开始尝试 → 首次确认，后续按偏好
  · LLM调用频率下降50%+（RL接管了大部分决策）
  · 新场景/新设备仍需LLM辅助
```

**关键优势**：随着RL模型成熟，系统**越来越不依赖LLM**，响应越来越快，成本越来越低。这跟原方案的"越用越懂你"方向一致，但机制更鲁棒——不是靠统计发现模式，而是靠真实用户反馈做强化学习。

## 4. 本地LLM优先策略

### 4.1 为什么本地优先

| 维度 | 云端API(OpenAI) | 本地LLM(Qwen2.5) |
|---|---|---|
| 响应延迟 | 500ms-2s(网络) | 50-200ms(本地) |
| 隐私风险 | 家庭对话数据上传 | 全本地，零泄露 |
| 成本 | $0.15/1K tokens | 0(电费而已) |
| 可用性 | 依赖网络 | 断网也能控制家 |
| 质量 | GPT-4o级 | Qwen2.5-7B级(略低) |

### 4.2 降级策略

```
意图消歧请求:
  1. 本地Qwen2.5-7B → 尝试消歧
  2. 置信度>0.7 → 执行（本地LLM够用）
  3. 置信度<0.7 → 降级到云端GPT-4o-mini → 再判断
  4. 云端也不确定 → 追问用户

设备离线时:
  · 本地LLM仍可用 → 生成"设备离线"提示
  · 不依赖云端 → 断网不失控
```

### 4.3 Mac Mini M4推理能力

- Qwen2.5-7B-Instruct: ~30 tokens/s (MLX优化) → 足够做意图消歧
- Qwen2.5-14B-Instruct: ~15 tokens/s → 更准确但稍慢
- 复杂场景编排 → 降级云端

## 5. MQTT骨干 + HA辅助

### 5.1 为什么MQTT优先

原方案用HA REST API作为唯一设备控制通道，问题：
- HA REST API是**请求-响应**模式，不适合事件驱动
- HA WebSocket虽可订阅状态，但**HA本身是瓶颈**——HA挂了全屋失控

MQTT骨干方案：
```
所有设备 ──→ MQTT Broker(Mosquitto) ──→ EventBus
                                      │
                    HA WebSocket ──────┤ (HA作为设备发现+初始接入工具)
                    传感器 ───────────┤ (直接MQTT推送)
                    用户指令 ─────────┤ (微信→EventBus)

HA的角色:
  · 设备发现: 新设备通过HA集成接入 → HA自动创建MQTT topic
  · 状态桥接: HA WebSocket → MQTT (HA作为"翻译器"而非"中枢")
  · 故障隔离: HA挂了 → MQTT仍可用 → 部分设备仍可控
```

### 5.2 MQTT Topic设计

```
harness-house/device/{room}/{device_type}/{device_id}/state    # 状态上报
harness-house/device/{room}/{device_type}/{device_id}/command   # 控制指令
harness-house/sensor/{room}/{sensor_type}/{sensor_id}/event     # 传感器事件
harness-house/user/command                                       # 用户指令
harness-house/rl/feedback                                        # RL反馈信号
harness-house/system/alert                                       # 系统告警
```

## 6. 安全分级体系（增强版）

### 6.1 三级安全（与原方案一致）

| 等级 | 策略 | 设备 |
|---|---|---|
| L1 | 自动执行 | 灯/风扇/窗帘/电视/猫粮机/晾衣杆/扫地机器人 |
| L2 | RL置信度>0.8自动，否则确认 | 空调温度/洗衣机/烘干机/热水器温度 |
| L3 | 永远确认 | 燃气热水器开关/门锁/监控 |

### 6.2 新增：场景安全审计

原方案缺少对**场景组合的安全审计**：

```
"睡觉了"场景 = 关灯(L1) + 关窗帘(L1) + 空调睡眠(L2) + 关热水器(L3)

问题: 场景中包含L3操作，整个场景不能一键执行
解决: 场景分级 = max(各设备等级) → "睡觉了"=L3场景 → 分步确认
  Step 1: 关灯+关窗帘+空调睡眠 → 自动执行(L1+L2部分)
  Step 2: "热水器也要关吗？" → 用户确认后执行(L3部分)
```

### 6.3 新增：设备离线处理

| 情况 | 处理 | 通知 |
|---|---|---|
| 设备离线+用户指令 | 返回"设备离线"提示 | "客厅灯当前离线，无法执行" |
| 设备离线+自动场景 | 跳过离线设备，执行其余 | "回家场景部分执行：空调✅ 窗帘❌(离线)" |
| 关键L3设备离线 | **主动告警** | "⚠️ 燃气热水器离线，请检查" |
| 全屋MQTT断连 | 切入安全模式 | 所有自动化暂停 + 告警 |

## 7. 边缘案例（原方案未覆盖）

### 7.1 多房间上下文切换

```
用户从书房走到客厅（传感器检测迁移）
· 30秒内收到"太亮了" → 当前区域=客厅（最近传感器位置）
· 5秒内收到"太亮了" → 当前区域=书房（刚离开的位置）
· 解决: 人在传感器加权：最近3秒的位置权重0.7，30秒前权重0.3
```

### 7.2 冬夏偏好自动切换

```
夏天: 空调制冷26度
冬天: 空调制热22度
· 同一句"热了" → 夏天=制冷, 冬天=可能暖气太足→调低制热温度
· 解决: RL状态空间包含season → 不同季节不同策略
· 不需要手动切换，RL自动学
```

### 7.3 客人模式

```
用户说"朋友来了" → 进入客人模式:
  · 监控不推送人脸识别结果(隐私)
  · 灯光/空调按主人偏好执行
  · 人体传感器不触发某些私人区域灯光(如卧室)
  · 客人离开后自动恢复
```

### 7.4 LLM API故障降级

```
云端API不可用:
  · 本地LLM接管所有意图消歧
  · 本地LLM也不可用(极端情况):
    · Rule Engine仍可用(90%日常操作不受影响)
    · 用户指令走关键词匹配(fallback)
    · 系统通知: "AI理解功能暂时降级，请用更明确的指令"
```

### 7.5 断电恢复策略

```
断电恢复后:
  · L1设备: 保持断电前状态(记忆恢复)
  · L2设备: 恢复到"安全默认"(空调不自动开→需确认)
  · L3设备: 恢复到"安全关闭"(热水器/门锁=关闭状态)
```

### 7.6 日程冲突

```
场景A: "回家"(开灯+开空调) 23:00触发
场景B: "睡觉"(关灯+关空调) 23:00触发
冲突 → 不执行 → 追问用户"你刚回家还是准备睡了？"
```

### 9.1 MVP核心验证

### 9.2 MVP功能

| 功能 | 范围 | 说明 |
|---|---|---|
| EventBus + MQTT | Mosquitto + Node.js Router | 事件驱动核心 |

| 本地LLM消歧 | Qwen2.5-7B(MLX) | 模糊指令处理 |
| 安全分级 | L1/L2/L3执行策略 | 与原方案一致 |
| 微信交互 | Hermes Agent接入 | 与原方案一致 |
| RL记录 | 用户反馈记录(不决策) | 冷启动只记录 |

| 云端LLM降级 | 本地LLM够用先不降级 |
