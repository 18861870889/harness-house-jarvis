## 1. 产品概述

### 1.1 产品名称
**Harness-House** — 用AI套住你的家

### 1.2 一句话定义
一个开源的AI智能家居控制系统，让用户用自然语言控制全屋设备，并持续学习主人的习惯，越用越懂你。

### 1.3 产品愿景
> 消灭智能家居的"App地狱"和"口令地狱"，让家真正听懂人话。

### 1.4 核心价值主张

| 现状（痛点） | Harness-House（解决方案） |
|---|---|
| 5+个App控制不同品牌设备 | **统一入口**：一个对话窗口控制全屋 |
| 必须背诵固定口令"打开客厅灯" | **自然语言**：说"太亮了"就知道调暗 |
| 场景绑定死，改场景要手动配置 | **自适应场景**：AI观察习惯自动生成 |
| 每个家人用法不同，系统一刀切 | **自进化**：越用越懂主人的偏好 |

## 2. 问题陈述

### 2.1 用户现状

- **设备生态碎片化**：米家App + 各厂商独立App，操作分散
- **交互僵硬**：必须用固定口令（"打开客厅灯"、"空调制冷26度"），无法表达模糊意图
- **场景死板**：预设场景绑定固定条件，调整需要手动配置，无法灵活组合
- **无个性化**：同一套规则服务所有人，不考虑个体习惯差异

### 2.2 目标用户

**主要用户**：自己（马佳峰）——有15+种智能设备的技术从业者  
**次要用户**：开源社区技术爱好者——有类似痛点想DIY解决方案  
**非目标用户**（当前阶段）：普通小白用户（需要更成熟的UX）

## 4. 交互设计

### 4.1 自然语言理解设计

**核心原则**：不要求用户说固定口令，AI从模糊表达中推断意图

| 用户说 | AI理解 | 执行 | 说明 |
|---|---|---|---|
| "太亮了" | 调暗当前区域灯光 | 灯光亮度→30% | 基于传感器推断用户所在区域 |
| "我回来了" | 回家场景 | 玄关灯开+客厅灯开+空调26度 | 触发预设+偏好场景 |
| "冷" | 升温 | 空调温度↑2度 | 基于偏好记忆决定升温幅度 |
| "睡觉了" | 睡眠场景 | 全屋灯关+窗帘关+空调睡眠模式 | 组合场景执行 |
| "书房灯开了吗？" | 状态查询 | 返回开关状态+亮度 | 查询而非操作 |
| "猫喂了吗？" | 查询喂食记录 | 返回上次喂食时间 | 跨设备状态查询 |

### 4.2 消歧机制

当意图模糊时，AI优先利用上下文推断，推断置信度低时才追问：

```
用户: "关灯"
AI推断链:
  1. 人在传感器 → 书房有人 → 置信度0.8 → 关书房灯 ✅直接执行

用户: "太热了"  
AI推断链:
  1. 当前区域=书房 → 书房空调执行 → 置信度0.6 → 偏低
  2. 追问: "书房空调调低2度，可以吗？" → 用户确认 → 执行
```

### 4.3 交互入口

| 通道 | 优先级 | 说明 |
|---|---|---|

## 5. 安全分级体系

### 5.1 设备安全等级定义

| 安全等级 | 定义 | 自动执行策略 | 设备举例 |
|---|---|---|---|
| **L1 安全** | 操作无安全风险，可即时执行 | ✅ 自动执行，无需确认 | 灯光、风扇、窗帘、电视、晾衣杆、猫粮机、扫地机器人 |
| **L2 注意** | 操作有舒适/能耗影响，误操作影响中等 | ⚠️ 首次执行需确认，后续按偏好自动 | 空调温度、洗衣机启动、烘干机、热水器温度调节 |
| **L3 危险** | 操作有安全风险，误操作可能造成危害 | 🔴 必须人工确认，不可自动执行 | 燃气热水器开关、门锁、监控操作 |

### 5.2 安全执行流程

```
用户指令 → Intent解析 → 设备识别 → 安全等级判定

L1设备: ──→ 直接执行 ──→ 结果反馈
L2设备: ──→ 首次? → 确认对话 ──→ 执行 ──→ 记录偏好 ──→ 后续自动
         ──→ 已有偏好? → 按偏好执行 ──→ 结果反馈
L3设备: ──→ 强制确认 ──→ 人工回复确认 ──→ 执行 ──→ 结果反馈
```

### 5.3 紧急停止机制

- 任何设备出现异常状态（如燃气温度异常）→ 自动告警+建议安全操作
- 用户可随时说"紧急停止" → 中止所有正在执行的自动化动作
- L3设备操作记录永久保留，可审计追溯

## 6. 自进化机制

### 6.1 进化维度

```
┌─────────────────────────────────────────┐
│           Self-Evolution Engine          │
│                                         │
│  ┌───────────┐  ┌───────────┐          │
│  │ Pattern   │  │ Preference │          │
│  │ Discovery │  │ Memory    │          │
│  └───────────┘  └───────────┘          │
│         │              │                │
│         ▼              ▼                │
│  ┌───────────┐  ┌───────────┐          │
│  │ Scene     │  │ Intent    │          │
│  │ Suggester │  │ Refinement│          │
│  └───────────┘  └───────────┘          │
└─────────────────────────────────────────┘
```

#### 维度1: 行为模式发现 (Pattern Discovery)

- **时间模式**：统计用户每天操作的时间分布，发现规律
  - 例："你最近7天都在23:00关客厅灯" → 建议"要不要自动化？"
- **关联模式**：发现设备操作的关联性
  - 例："你开空调时总是同时关窗帘" → 建议"创建'凉爽'场景"
- **条件模式**：发现操作与环境条件的关系
  - 例："室外温度>30°C时你总是开空调制冷26度" → 建议条件自动化

#### 维度2: 偏好记忆 (Preference Memory)

- **数值偏好**：空调喜欢26度制冷、卧室灯喜欢30%亮度
- **场景偏好**：睡前习惯关窗帘+空调睡眠模式+灯全关
- **区域偏好**：在书房时习惯开台灯而非主灯
- **渐进学习**：每次用户纠正AI推断，偏好权重更新

#### 维度3: 意图消歧优化 (Intent Refinement)

- **初始**："太亮了" → AI追问"哪个房间？"
- **学习后**："太亮了" + 人在传感器=书房 → AI直接调书房灯
- **纠正记录**：用户说"不是书房，是客厅" → 更新区域-意图映射

#### 维度4: 场景建议 (Scene Suggester)

- 基于发现的模式，主动建议创建场景自动化
- 用户可通过对话确认/修改/拒绝建议
- 确认的场景自动进入Orchestration Layer定时执行

### 6.2 进化数据存储

| 数据类型 | 存储内容 | 用途 |
|---|---|---|
| ActionLog | 每次设备操作的指令/时间/设备/结果 | Pattern Discovery原始数据 |
| PreferenceDB | 用户的数值/场景/区域偏好 | 执行时的偏好参考 |
| PatternStore | 发现的行为模式+置信度 | Scene建议+条件自动化 |
| IntentMap | 模糊意图→具体操作的映射表 | Intent消歧优化 |
| SafetyRecord | L2/L3设备的操作确认记录 | 偏好学习+安全审计 |

### 7.4 HomeAssistant接入方案

**核心策略**：HA作为设备统一接入层，AI Agent通过HA REST API控制所有设备

```
米家设备 ──→ HA Xiaomi Miot Auto集成 ──→ HA REST API ──→ AI Agent
涂鸦设备 ──→ HA Tuya集成           ──→ HA REST API ──→ AI Agent
Zigbee设备──→ HA Zigbee2MQTT       ──→ HA REST API ──→ AI Agent
其他设备  ──→ HA相应集成/自定义组件 ──→ HA REST API ──→ AI Agent
```

HA优势：
- 已有1000+官方集成，覆盖主流品牌
- 米家Xiaomi Miot Auto集成可接入大部分米家设备
- REST API + WebSocket提供完整控制+状态查询
- 本地运行，数据隐私

## 9. MVP范围定义

### 9.1 MVP目标

**证明核心假设**：AI Agent + HomeAssistant可以实现自然语言智能家居控制，比App+口令模式体验更好。

### 9.2 MVP功能范围（P0）

| 功能 | 实现范围 | 交付标准 |
|---|---|---|
| 自然语言控制 | L1设备控制+状态查询 | "太亮了"→调暗灯光，"灯开着吗"→返回状态 |
| 设备接入 | 米家设备通过HA接入 | 灯光、空调、风扇、窗帘、传感器可控制可查询 |
| 安全分级 | L1自动/L2确认/L3禁止自动 | 燃气热水器操作必须确认，灯光可直接执行 |
| 微信交互 | 微信Bot作为交互入口 | 发消息→AI响应→设备执行→结果反馈 |
| 偏好记录 | 手动设置偏好值 | "记住我喜欢空调26度"→写入偏好库 |

### 9.3 MVP不做

| 不做 | 理由 |
|---|---|
| 自动模式发现 | 需要积累足够ActionLog数据，MVP阶段数据量不足 |
| 语音交互 | 需额外集成语音识别，增加复杂度 |
| Web Dashboard | 可视化是体验提升，非核心验证 |
| 多用户区分 | MVP只服务单一用户 |
| 非米家设备接入 | 先做好一个生态，再扩展 |

### 9.4 MVP技术栈精简

```

设备控制: HomeAssistant REST API
对话入口: 微信Bot (通过Hermes Agent接入)
数据存储: SQLite (本地)
部署: Mac Mini本地运行
```

## 12. 风险与应对

| 风险 | 影响 | 应对措施 |
|---|---|---|
| HA集成不支持某些米家设备 | 无法控制部分设备 | 提前测试Xiaomi Miot Auto覆盖范围；不支持的自写HA组件 |
| LLM意图理解不准 | 误操作设备 | 安全分级兜底；置信度低时追问；L3永不自动 |
| 微信Bot限频/封号 | 交互通道中断 | 双通道(微信+Telegram)互备；Web Dashboard兜底 |
| 本地部署HA不稳定 | 设备控制中断 | Docker化+自动重启+健康检查 |
| AI过度自动化 | 用户失控感 | 所有自动化必须用户确认；紧急停止机制；进化建议而非强制 |
| 数据隐私 | 家庭行为数据泄露 | 全本地存储+本地LLM可选；不上传云端 |
