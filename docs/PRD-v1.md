## 2. 问题陈述

### 2.1 用户现状

- **设备生态碎片化**：米家App + 各厂商独立App，操作分散
- **交互僵硬**：必须用固定口令，无法表达模糊意图

- **过度依赖云**：现有AI方案断网即瘫痪

### 2.2 目标用户

**主要**：自己（马佳峰）——有15+种智能设备的技术从业者  
**次要**：开源社区技术爱好者  
**非目标**（当前）：普通小白用户

## 3. 架构设计

### 3.1 事件驱动流水线架构

**核心洞察**：智能家居90%的操作是事件驱动的（传感器触发、定时触发），只有10%是用户手动指令。AI Agent不应该是"中心管家"，而是管道中的一个智能节点。

```
┌─────────────────────────────────────────────────────────┐
│                    Event Sources                         │
│  传感器 / 定时 / 用户指令 / 设备状态变更 / 天气API       │
└──────────────────────┬──────────────────────────────────┘
                       │
                ┌──────▼──────┐
                │   EventBus   │
                │  (MQTT +    │
                │  Router)     │
                └──────┬──────┘
                       │
          ┌────────────┼────────────┐
          │            │            │
   ┌──────▼──────┐ ┌──▼───────┐ ┌──▼──────┐
   │ Rule Engine │ │ LLM Agent│ │ RL Learner│
   │ (确定性规则) │ │(模糊消歧)│ │(进化决策) │
   │  90%流量    │ │  10%流量  │ │  渐进接管 │
   └──────┬──────┘ └──┬───────┘ └──┬──────┘
          │            │            │
          └────────────┼────────────┘
                       │
                ┌──────▼──────┐
                │   Safety    │
                │  Guardian   │
                │ (L1/L2/L3) │
                └──────┬──────┘
                       │
                ┌──────▼──────┐
                │  Device Bus │
                │ (MQTT骨干 + │
                │  HA桥接)    │
                └──────┬──────┘
                       │
                ┌──────▼──────┐
                │  Physical   │
                │  Devices    │
                └──────┴──────┘
```

### 3.2 路由策略

```python
def route_event(event):
    # 已有确定性规则 → 直接走Rule Engine，零延迟零成本
    if event.match_known_rule():
        return RuleEngine.handle(event)
    
    # RL已学会的映射 → 自动执行(渐进接管)
    learned = RL.lookup(event.state)
    if learned and learned.confidence > 0.6:
        return learned  # RL接管
    
    # 模糊指令 → 路由到LLM消歧
    if event.type == "user_command":
        return LLMAgent.disambiguate(event)
    
    # 默认走Rule Engine
    return RuleEngine.handle_default(event)
```

**关键优势**：
- Rule Engine覆盖90%日常操作 → **零LLM成本、零延迟**
- RL Learner渐进接管 → **LLM调用越来越少**
- 本地LLM优先处理模糊指令 → **隐私不泄露**
- 断网时Rule Engine+本地LLM仍可用 → **不失控**

## 5. 交互设计

### 5.1 自然语言理解

**核心原则**：不要求固定口令，AI从模糊表达推断意图

| 用户说 | AI理解 | 执行 | 推断依据 |
|---|---|---|---|
| "太亮了" | 调暗当前区域灯光 | 灯光→30% | 人在传感器推断位置 |
| "我回来了" | 回家场景 | 玄关灯+客厅灯+空调 | 触发预设+偏好 |
| "冷" | 升温 | 空调温度↑2度 | 偏好记忆+季节 |
| "睡觉了" | 睡眠场景 | 灯关+窗帘关+空调睡眠 | 组合场景 |
| "书房灯开了吗？" | 状态查询 | 返回开关+亮度 | 查询而非操作 |
| "猫喂了吗？" | 查询喂食记录 | 返回上次喂食时间 | 跨设备查询 |

### 5.2 消歧机制

```
"关灯" → 人在传感器=书房 → 置信度0.8 → L1设备 → 直接执行
"太热了" → 当前=书房 → 置信度0.6 → L2设备 → 确认后执行
"关热水器" → L3设备 → 永远确认 → 不论置信度
```

### 5.3 RL反馈交互

```
AI: 书房灯调到30%？（基于RL学习）
用户: 嗯                     → RL奖励+1 → 强化此映射
用户: 不是书房，是客厅       → RL惩罚-0.5 + 更新映射 → 下次更准
用户: 太暗了                 → RL惩罚-1 → 纠正亮度偏好
用户沉默接受                 → RL奖励+0.5 → 轻度正反馈
```

| 通道 | 优先级 | 说明 |
|---|---|---|
| 微信Bot | P0 | 主力通道，用户日常习惯 |
| Telegram Bot | P1 | 开源社区通道 |
| Web Dashboard | P1 | 可视化+场景管理 |
| 语音 | P2 | 长期目标 |

## 6. 安全分级体系

### 6.1 设备安全等级

| 等级 | 定义 | 执行策略 | 设备 |
|---|---|---|---|
| **L1 安全** | 无安全风险 | ✅ 自动执行 | 灯光、风扇、窗帘、电视、晾衣杆、猫粮机、扫地机器人 |
| **L2 注意** | 中等影响 | ⚠️ RL置信度>0.8自动，否则确认 | 空调温度、洗衣机、烘干机、热水器温度 |
| **L3 危险** | 有安全风险 | 🔴 永远确认 | 燃气热水器开关、门锁、监控 |

### 6.2 场景安全审计

**关键新增**：场景的等级 = 场景中最高设备等级

```
"睡觉了" = 关灯(L1) + 关窗帘(L1) + 空调睡眠(L2) + 关热水器(L3)
场景等级 = L3 → 分步执行:
  Step 1: L1+L2部分自动执行（灯+窗帘+空调）
  Step 2: "热水器也要关吗？" → 用户确认后执行L3部分
```

### 6.3 设备离线处理

| 情况 | 处理 | 通知 |
|---|---|---|
| L1离线+用户指令 | 返回"设备离线" | "客厅灯离线，无法执行" |
| 设备离线+场景 | 跳过离线，执行其余 | "回家场景部分：空调✅ 窗帘❌(离线)" |
| L3设备离线 | **主动告警** | "⚠️ 燃气热水器离线，请检查" |
| MQTT断连 | 安全模式 | 全部自动化暂停+告警 |

### 6.4 紧急停止

- "紧急停止" → 中止所有自动化
- L3操作永久记录，可审计追溯
- 断电恢复：L1恢复断电前状态 / L2恢复安全默认 / L3恢复安全关闭

## 7. 自进化机制

### 7.1 三层进化引擎

```
┌─────────────────────────────────────┐
│        Evolution Engine              │
│                                     │
│  ┌───────────┐  ┌───────────────┐  │
│  │   RL Core │  │ Preference    │  │
│  │  (强化学习)│  │ Memory       │  │
│  │  用户反馈  │  │ (偏好存储)    │  │
│  │  =奖励信号 │  │  数值/场景/  │  │
│  └─────┬─────┘  │ 区域偏好     │  │
│        │        └────────┬──────┘  │
│        │                 │         │
│  ┌─────▼─────────────────▼──────┐  │
│  │      Pattern Discovery       │  │
│  │   (统计模式发现)              │  │
│  │   时间/关联/条件模式          │  │
│  │   → 建议场景 → RL验证       │  │
│  └──────────────────────────────┘  │
└─────────────────────────────────────┘
```

**三层协同**：
1. **RL Core**（主动进化）：用户反馈做奖励/惩罚 → 渐进接管决策
2. **Preference Memory**（被动存储）：记住数值偏好 → 执行时参考
3. **Pattern Discovery**（发现引擎）：统计发现规律 → 建议场景 → 交给RL验证

### 7.2 RL状态空间

```python
state = {
    "location": 人在传感器推断位置,      # 书房/客厅/厨房/玄关
    "time_slot": 时间段,                 # morning/afternoon/evening/night
    "temperature_indoor": 室内温度,
    "temperature_outdoor": 室外温度,
    "light_level": 当前区域光照,
    "day_of_week": 星期几,
    "season": 季节,                      # ← 关键：冬夏偏好自动切换
    "recent_actions": 最近5分钟操作序列,
}
```

### 7.3 RL奖励信号

| 用户行为 | 奖励值 | 含义 |
|---|---|---|
| "嗯"/"好"/"开" | +1.0 | 决策正确，强化 |
| 沉默接受 | +0.5 | 可能正确 |
| 纠正"不是书房" | -0.5 + 新映射 | 消歧偏差 |
| "不对"/"取消" | -1.0 | 决策错误 |
| 主动指令(AI没自动) | 0 | 中性记录 |

### 7.4 进化路径

```
Week 1-2 (冷启动):
  · 所有决策走 Rule Engine + LLM消歧
  · RL只做记录，不做决策
  · 用户反馈积累训练数据

Week 3-4 (学习期):
  · RL在L1设备上尝试决策（低置信仍走LLM）
  · 高置信(>0.6)的L1设备直接执行

Month 2+ (成熟期):
  · RL在L1达到>80%准确 → 自动执行
  · RL开始接管L2 → 首次确认后自动
  · LLM调用频率下降50%+
  · 新场景/新设备仍需LLM辅助
```

### 7.5 进化数据存储

| 数据类型 | 存储内容 | 技术 |
|---|---|---|
| ActionLog | 每次操作指令/时间/设备/结果 | TimescaleDB(时序) |
| PreferenceDB | 数值/场景/区域偏好 | SQLite |
| RLModel | 强化学习策略模型 | pickle/ONNX |
| PatternStore | 行为模式+置信度 | SQLite |
| SafetyRecord | L2/L3操作确认记录 | SQLite(永久) |

### 8.2 技术选型

| 层 | 技术 | 选型理由 |
|---|---|---|
| EventBus/Router | Node.js + MQTT(Mosquitto) | 异步IO天生优势；WebSocket/MQTT原生支持 |
| Rule Engine | Node.js | 确定性规则快速执行，零延迟 |
| LLM Agent | 本地Qwen2.5-7B(MLX) + 云端降级(GPT-4o-mini) | 隐私+低延迟+零成本；复杂场景降级云端 |
| RL Learner | Python(Stable-Baselines3) | RL生态成熟；独立进程 |
| Safety Guardian | Node.js | 路径拦截器，轻量快速 |
| 设备接入 | MQTT骨干 + HA桥接 | HA做接入翻译，MQTT做实时骨干 |
| 时序数据 | TimescaleDB | 专为时序优化；RL训练+Pattern分析 |
| 偏好数据 | SQLite | 轻量偏好+安全记录 |
| 对话入口 | Hermes Agent(微信) + Telegram Bot | 利用现有基础设施 |
| 部署 | Docker Compose on Mac Mini M4 | 本地推理+全栈容器化 |

### 8.4 HA→MQTT桥接

```
米家设备 → HA Xiaomi Miot Auto → HA WebSocket → MQTT桥接 → EventBus

HA角色: 设备发现+接入翻译(不是中枢)
MQTT角色: 实时通信骨干(HA挂了MQTT仍可用)
```

### 8.5 本地LLM降级策略

```
意图消歧请求:
  1. 本地Qwen2.5-7B → 置信度>0.7 → 执行
  2. 置信度<0.7 → 降级云端GPT-4o-mini
  3. 云端也不确定 → 追问用户

断网时:
  · 本地LLM仍可用 → 做消歧
  · Rule Engine仍可用 → 90%操作不受影响
  · 系统通知: "AI理解功能降级，请用更明确指令"
```

## 10. 边缘案例处理

| 场景 | 处理策略 |
|---|---|
| **多房间切换** | 人在传感器加权：最近3秒位置权重0.7，30秒前权重0.3 |
| **冬夏偏好切换** | RL状态含season → 同一句"热了"夏天制冷、冬天调低制热 |
| **客人模式** | "朋友来了" → 监控不推人脸、私人区域灯不自动开 |
| **LLM API故障** | 本地LLM接管 → Rule Engine兜底 → 关键词匹配fallback |
| **设备离线** | L1返回提示/L3主动告警/场景跳过离线执行其余 |
| **断电恢复** | L1恢复断电前/L2安全默认/L3安全关闭 |
| **场景冲突** | 同一时间两个场景 → 不执行 → 追问用户意图 |
| **摄像头隐私** | L3设备不发送图像数据到云端LLM |

## 11. MVP范围

### 11.1 MVP核心验证

**验证假设**：事件驱动+本地LLM+RL记录的架构，能否比App+口令模式体验更好，且成本更低。

### 11.2 MVP功能

| 功能 | 范围 | 交付标准 |
|---|---|---|
| EventBus+MQTT | Mosquitto + Node.js Router | 事件流正常流转 |
| Rule Engine | 10条预设规则 | 传感器→灯等日常操作零延迟 |
| 本地LLM消歧 | Qwen2.5-7B(MLX) | "太亮了"→正确调暗 |
| 安全分级 | L1/L2/L3策略 | 燃气必须确认 |
| 微信交互 | Hermes Agent接入 | 发消息→响应→执行→反馈 |
| RL记录 | 用户反馈记录(不决策) | 冷启动只做记录 |
| 设备接入 | HA→MQTT桥接(5米家设备) | 灯+空调+风扇+窗帘+传感器 |

### 11.3 MVP不做

| 不做 | 理由 |
|---|---|
| RL决策 | 冷启动数据不足，先记录 |
| 场景编排 | Rule Engine覆盖核心场景 |
| Telegram/Web | MVP只用微信 |
| 云端LLM降级 | 本地LLM够用先不降级 |

### 11.4 开发周期

| 阶段 | 内容 | 时间 |
|---|---|---|
| Phase 0 | Mosquitto + HA→MQTT桥接 + 5设备接入 | 2-3天 |
| Phase 1 | EventBus + Rule Engine + 10规则 | 2-3天 |
| Phase 2 | 本地Qwen2.5-7B + 消歧接口 | 2-3天 |
| Phase 3 | 微信Bot接入 + RL反馈记录 | 2天 |
| Phase 4 | 安全分级 + 测试 | 2天 |
| **总计** | | **~2周** |

## 14. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| HA集成不支持某设备 | 无法控制 | 提前测试；不支持的自写组件 |
| 本地LLM质量不够 | 消歧不准 | 云端降级兜底；安全分级拦截 |
| RL冷启动慢 | 前期进化慢 | Rule Engine+LLM先行；RL渐进接管 |
| 微信Bot限频 | 交互中断 | Telegram互备；Web兜底 |
| HA挂了 | 部分设备失控 | MQTT骨干独立运行；HA只是桥接 |
| AI过度自动化 | 用户失控 | 紧急停止；进化建议而非强制 |
| 数据隐私 | 行为泄露 | 本地LLM+本地存储；L3不上传 |
| 断网断电 | 系统瘫痪 | Rule Engine本地运行；断电安全恢复 |
